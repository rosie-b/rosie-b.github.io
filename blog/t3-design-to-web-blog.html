#6 Design to Web

Responsive site

What is it and why is it important?

A responsive site is where the site layout, sizes, look, feel and (if appropriate) function all change depending on the size of device the user is accessing the site from.

This is important because users now access the internet from many different sized screens, which also vary from being touch screen to keyboard/mouse driven. And that means that the impact of a site being only built for one of these mediums is big and could be the difference between someone using your site/product or going elsewhere.

Mobile first design

What is it and why is it important?

This is where the site is built on the assumption that the user will access it from a mobile device, with all code written for this and then the additional queries relating to bigger screens are added on.

This is important for styling, usability and performance. For instance, a smaller screen will need to have the content be visible (don't want to be zooming in/out to navigate the page and read the content). It will also be touch screen, so buttons about the size of a finger will be more user friendly than a hyperlink from a word/phrase. From a performance perspective, the amount of processing power a mobile device has will likely be lower than a desktop. By coding for mobile first, anything dictating the look/content/functionality for larger decives will only be accessed by those larger devices. So the smaller mobile device doesn't need to sift through all that code just to get to the relevant bits.

Frameworks

Pros:

  • Don't need to "reinvent the wheel" each time a new project is started.
  • Good way to learn what is possible or a differnt way of approaching something.
  • Consistency of approach, naming convensions, methods makes it easier for a developer to pick up pieces of work from others and keep on going with it.
  • Reduced build time, especially if familiar with the framework being used.

Cons:

  • Learning a new framework can take more time than writing from scratch (especially for a small project).
  • Could become reliant on a framework and limit the amount of exploration/play that could lead to a better way of coding an element.
  • Dependency on a framework that has been build by someone else.
  • Potential for reduced performance if the framework has more code than required.

Wireframes

What is it and why do we use it?

These are the basic plans that we draw up before starting to build a site, mapping out the basic structure and elenments of the page. We use the wireframe to make it easier to understand how to best code the site (containers, sections, divisions, headers etc). And also to plan how the site may look/be structured in both mobile and desktop view. These drawings are then a useful visual tool to use as coding progreses.

Implementing the wireframe

Challenges in my wireframe came from not having the time or skills (yet) to get the layout/functionality that I really want (e.g. working out the best ways to organise the divisions to get a certain look for the blog page navigation - I couldn't get this to look the way I designed it).

Along with this, came the challenge of knowing when I've tried enough different angles to get something to look like my wireframe and when I just need to accept that it will look different and move on.

I can imagine that this will be a key challenge in bringing designs to life in future projects, with the key to this being able to understand what is possible in the time/with the skills available. Then, what could be possible with more time and exploration. And keeping the communication live throughout to make sure that any deviations from the wireframe are accepetable.
